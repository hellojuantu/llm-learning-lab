# Course Design Philosophy

这份课程设计采用的是“理解台阶”，不是“题海”。

## 核心原则

题的数量不固定。真正重要的是：每道题都必须让学生跨过一个具体障碍。

一节课的作业应该像这样生长：

```text
完整例子
-> 照着改一个数字
-> 换一个符号/分支/维度
-> 遇到边界情况
-> 抽象成函数或规则
-> 接入项目主线
-> 修一个故意坏掉的版本
```

这和只写“知识点说明”不同。学生必须在一个可运行工件里操作、犯错、测试、修正、复盘。

## 课程里的主线

micrograd 的学习主线不是“把源码看完”，而是理解这条链：

```text
输入数据
-> 前向计算 data
-> 计算 loss
-> backward 得到每个参数的 grad
-> 按 -grad 更新参数
-> loss 下降
```

每一节都要接回这条链。

```text
数学课：解释 grad 到底是什么
Value 使用课：观察 data/grad/backward 的行为
源码课：解释 micrograd 为什么能算 grad
TinyValue 作业：自己造一个小 autograd
Neuron/MLP：把标量 Value 组合成神经网络
训练循环：把 grad 用来更新参数
分类边界：把训练结果画出来
PyTorch 桥接：看工业库封装了 micrograd 的哪些动作
Debug 课：训练失败时能定位问题
Mini Project：把整条链讲清楚
```

## 设计验收

不要问“这节有多少题”。要问：

```text
这一组题拆掉了哪个理解障碍？
下一题是否复用了上一题？
学生填错后，测试能不能指出错在数学、Python、图结构还是训练循环？
Debug Lab 是否来自真实学习错误？
最后的项目题是否接回 micrograd 主线？
学生能不能用自己的话解释这节为什么存在？
```

如果答案是否定的，这节就还没有设计完。
